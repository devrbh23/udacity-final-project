1. SYSTEM OVERVIEW
----------------------------------------------------------------
 
The Munder Difflin Multi-Agent System automates three core
business workflows for a fictional paper supply company:
 
  1. Inventory management and restocking
  2. Quote generation with bulk discounts
  3. Order fulfillment and transaction recording
 
The system processes incoming customer requests end-to-end,
from checking stock through to recording revenue, using a
pipeline of four coordinated AI agents built on the smolagents
framework.
 
----------------------------------------------------------------
2. WORKFLOW DIAGRAM EXPLANATION
----------------------------------------------------------------
 
The workflow_diagram.svg submitted with this project shows a
vertical pipeline of four agents, reading top to bottom:
 
  [Customer Request]
         |
  [Inventory Agent] — tools on the LEFT side of the diagram
         | (JSON)
  [Quote Agent] — tools on the RIGHT side of the diagram
         | (JSON)
  [Order Agent] — tools on the LEFT side of the diagram
         | (JSON)
  [Orchestrator] — no tools; synthesis only
         |
  [Customer Response]
 
At the bottom of the diagram, all three worker agents connect
(dashed lines) to the SQLite database, representing read/write
operations. A discount tier legend at the bottom shows the
three thresholds (>$500, >$1000, >$5000) visually.
 
The architecture was designed with three guiding principles:
 
  SEPARATION OF CONCERNS: Each agent has exactly one job.
  The Inventory Agent never generates prices; the Quote Agent
  never touches transactions; the Order Agent never checks
  historical quotes. This makes each agent independently
  testable and replaceable without affecting the others.
 
  SEQUENTIAL CONTEXT CHAINING: Each agent's JSON output
  becomes part of the next agent's input prompt. This means
  the Quote Agent always knows the current stock status before
  pricing, and the Order Agent always has the exact approved
  quote before executing transactions. No agent operates on
  stale or assumed information.
 
  ORCHESTRATOR AS SYNTHESISER: The Orchestrator was
  deliberately given no tools. Its only job is to read the
  JSON outputs from all three workers and produce a coherent,
  customer-friendly narrative. Separating synthesis from
  execution prevents the Orchestrator from accidentally
  triggering duplicate transactions or interfering with the
  worker agents' decisions.

2. AGENT ARCHITECTURE (4 Agents)
----------------------------------------------------------------
 
AGENT 1 — Inventory Agent
  Role    : Check stock levels; restock items that are low
  Class   : smolagents ToolCallingAgent
  Tools   : CheckInventoryTool, GetFullInventoryTool,
            RestockItemTool
  Inputs  : Customer request text, date
  Outputs : JSON summary of items checked and any restock orders
  Logic   : If any requested item has fewer than 100 units OR
            less than the requested quantity, automatically
            places a restock order (minimum 500 units).
            Cash-aware: never orders more than the available
            cash balance allows.
 
AGENT 2 — Quote Agent
  Role    : Generate an itemised, discount-aware price quote
  Class   : smolagents ToolCallingAgent
  Tools   : LookupPastQuotesTool, GetItemPriceTool,
            GetCatalogueTool
  Inputs  : Customer request, date, inventory summary
  Outputs : JSON quote with line items, discount rate, total
  Logic   : Searches historical quote data for pricing context,
            maps customer descriptions to exact catalogue items
            (fuzzy matching), and applies tiered bulk discounts:
              - Orders > $500  → 5% discount
              - Orders > $1000 → 10% discount
              - Orders > $5000 → 15% discount
 
AGENT 3 — Order Agent
  Role    : Execute confirmed sale transactions
  Class   : smolagents ToolCallingAgent
  Tools   : ProcessSaleTool, GetCashBalanceTool,
            GetSupplierLeadTimeTool
  Inputs  : Customer request, date, approved quote
  Outputs : JSON transaction summary with statuses and revenue
  Logic   : Verifies cash availability, processes each line item
            as a separate sale transaction (one DB call per
            item), skips items with insufficient stock and
            reports them clearly.
 
AGENT 4 — Orchestrator
  Role    : Coordinate agents 1-3; synthesise customer response
  Class   : smolagents ToolCallingAgent (no tools, LLM only)
  Tools   : None (reasoning and synthesis only)
  Inputs  : Customer request + outputs from all three agents
  Outputs : Plain-English customer-facing response
  Logic   : Runs Agents 1 to 3 in sequence, passing context
            forward at each step. Synthesises all results into
            a single professional response including quote
            details, fulfillment status, and delivery dates.
 
----------------------------------------------------------------
3. AGENT COMMUNICATION PATTERN
----------------------------------------------------------------
 
All agents communicate via structured text (JSON passed as
prompt context). There is no shared memory or message bus.
 
Pipeline flow per request:
 
  Customer Request
       |
       v
  [Inventory Agent] --> Inventory Summary (JSON)
       |
       v
  [Quote Agent] --> Quote Details (JSON)
       |
       v
  [Order Agent] --> Transaction Summary (JSON)
       |
       v
  [Orchestrator] --> Customer Response (plain English)
 
Each agent receives the outputs of all previous agents as
context in its prompt. This creates a chain where each agent
has full awareness of prior decisions.

4. TOOL DESIGN DECISIONS
----------------------------------------------------------------
 
Each smolagents tool is a subclass of smolagents.Tool with:
  - name, description, inputs, output_type class attributes
  - a forward() method implementing the actual logic
 
Tool implementations are thin wrappers around the database
utility functions provided in the starter code. Each tool:
 
  - Returns a JSON string so it can be passed back to the LLM
    as a tool result in the smolagents message format
  - Validates constraints before committing (e.g. stock check
    before processing a sale, cash check before restocking)
  - Handles errors gracefully and returns informative error
    messages in the JSON payload
 
The GetItemPriceTool includes fuzzy matching: if the LLM
passes a description like "heavy cardstock" instead of the
exact catalogue name "250 gsm Cardstock", the tool finds the
closest match so quotes still succeed.
 
The LookupPastQuotesTool accepts a comma-separated string for
search terms (rather than a list) because smolagents tool
inputs must be primitive JSON types; the tool splits the
string internally before calling search_quote_history().
 
----------------------------------------------------------------
5. THE smolagents TOOL-USE LOOP
----------------------------------------------------------------
 
Each worker agent is a smolagents ToolCallingAgent. The
framework handles the tool-use loop automatically:
 
  1. The agent receives a task prompt
  2. smolagents sends the prompt + tool schemas to the LLM
  3. The LLM returns tool calls; smolagents executes them
  4. Results are appended to the conversation automatically
  5. The loop continues until the LLM calls final_answer()
  6. The final_answer content is returned to our code
 
max_steps is set per agent (8 for Inventory and Quote, 10 for
Order) to prevent infinite loops on complex requests.
 
The system_prompt for each agent is injected via:
  prompt_templates={**EMPTY_PROMPT_TEMPLATES, "system_prompt": ...}
This preserves smolagents' required planning, managed_agent,
and final_answer template keys while customising only the
system instruction for each agent's role.

6. FRAMEWORK CHOICE
----------------------------------------------------------------
 
This project uses smolagents (version >=1.24.0) as the agent
orchestration framework, which is one of the three explicitly
recommended frameworks for this project. The requirements.txt
includes smolagents as a dependency.
 
smolagents was selected for the following reasons:
 
  a) Tool-first design: smolagents' Tool subclass pattern maps
     cleanly onto the existing database utility functions. Each
     utility (get_stock_level, create_transaction, etc.) was
     wrapped as a typed Tool subclass with name, description,
     inputs, output_type, and a forward() method.
 
  b) OpenAI-compatible model support: smolagents provides
     OpenAIServerModel, which connects directly to the Vocareum
     OpenAI-compatible proxy endpoint (openai.vocareum.com/v1)
     without any modification to the request format.
 
  c) Automatic tool-use loop: ToolCallingAgent manages the
     LLM <-> tool call <-> result loop internally, removing the
     need to manually implement message history management or
     tool dispatch logic that would otherwise require hundreds
     of lines of boilerplate.
 
  d) Minimal dependencies: smolagents adds only one package
     beyond the project's existing requirements.txt, keeping
     the environment simple and reproducible.
 
The model used is gpt-4o-mini. The API key is loaded from the
UDACITY_OPENAI_API_KEY environment variable (with OPENAI_API_KEY
as fallback) using python-dotenv.

7. TEST RUN RESULTS — ANALYSIS
----------------------------------------------------------------
 
Test period    : 2025-04-01 to 2025-04-17
Total requests : 20
Customer types : office managers, hotel managers, school boards,
                 city clerks, restaurant managers, event planners
Starting cash  : $45,059.70
Ending cash    : $49,706.20
Net change     : +$4,646.50 (net revenue after restock costs)
 
STRENGTH 1 — Correct tiered discount application at scale
  The Quote Agent correctly applied the three discount tiers
  across all 20 requests covering diverse customer types and
  event contexts (ceremonies, conferences, parades, concerts,
  exhibitions, restaurant supply). Evidence from test_results.csv:
  Request 6 (city clerk, large parade order) produced the
  largest single-request revenue gain, consistent with a bulk
  order exceeding the $5,000 threshold and receiving the 15%
  discount. Requests 1-5 (smaller ceremony and office orders)
  received either no discount or the 5% tier, matching their
  lower order sizes. This demonstrates the discount logic
  scaled correctly from small to large orders without manual
  intervention or hard-coded per-customer rules.
 
STRENGTH 2 — Graceful partial fulfillment with clear communication
  Several requests involved items that were either out of stock
  or pending a restock delivery that fell after the customer's
  required date. In every such case, the Order Agent's
  ProcessSaleTool verified stock via get_stock_level() before
  committing each transaction, returned a structured
  {"status": "failed", "reason": "..."} JSON payload for
  unavailable items, and the Orchestrator translated this into
  a plain-English explanation in the customer response. No
  request produced an unhandled exception or a silent failure
  where an item was simply omitted without explanation. This
  is a key production-readiness strength: the system degrades
  gracefully rather than failing entirely when stock is short.
 
STRENGTH 3 — Delivery date propagation through the full pipeline
  All 20 customer responses included a delivery date or
  delivery timeline. The RestockItemTool called
  get_supplier_delivery_date() on every restock order and
  embedded the result in its JSON output. This date then
  appeared in the Quote Agent's context (which included the
  inventory summary) and propagated to the Order Agent and
  Orchestrator without any of those agents being explicitly
  instructed to carry it forward. This demonstrates that the
  chained JSON context design successfully passes critical
  logistics information end-to-end — a property that emerged
  from the architecture rather than being explicitly engineered
  at each step.
 
STRENGTH 4 — Cash integrity across all 20 requests
  No request produced a negative cash balance. The
  RestockItemTool checks get_cash_balance() before every
  supplier order and automatically reduces the restock
  quantity if the full order would exceed available funds.
  This constraint held across the entire test run: early
  requests (1-5) showed small net cash decreases as restock
  costs slightly exceeded sale revenues from small orders,
  while later large-volume requests restored and grew the
  balance. The ending cash of $49,706.20 (higher than the
  starting $45,059.70) confirms the business was net-profitable
  across the test period, with no overspending on inventory.
 
STRENGTH 5 — Multi-item order handling across diverse contexts
  Requests ranged from single-item orders (e.g. Request 1:
  A4 paper for a ceremony) to complex multi-item orders
  (e.g. school board ordering 8+ different supplies for a
  semester). The Order Agent correctly issued one
  ProcessSaleTool call per line item in every case, ensuring
  each transaction was independently recorded and individually
  verifiable in the database. No line items were silently
  merged or skipped in multi-item orders.
 
AREA FOR IMPROVEMENT — Occasional discount reasoning variance
  A small number of responses described discount percentages
  that did not precisely match the tier thresholds. This is a
  runtime LLM reasoning issue: gpt-4o-mini occasionally
  approximates the discount calculation rather than applying
  the exact threshold rule from the system prompt. The fix —
  a deterministic CalculateDiscountTool — is described in
  Section 11 (Improvement 2). This is the most significant
  quality gap identified in the test run.

11. FUTURE IMPROVEMENT SUGGESTIONS
----------------------------------------------------------------
 
IMPROVEMENT 1 — Persistent Agent Memory with Quote Embeddings
--------------------------------------------------------------
Currently the quote_agent searches historical quotes using
simple keyword matching via search_quote_history(). A more
powerful approach would be to embed each historical quote as a
vector (using OpenAI's text-embedding-3-small model) and store
them in a vector database (e.g. ChromaDB or pgvector). When a
new customer request arrives, the quote_agent would embed the
request and retrieve the top-K most semantically similar past
quotes rather than just keyword matches. This would improve
pricing accuracy for complex or unusual requests where keyword
overlap is low (e.g. a request for "biodegradable event
supplies" would find quotes for "eco paper plates" even without
exact keyword overlap).
 
Implementation steps:
  a) Add an embedding step to init_database() that vectorises
     all rows in the quotes table on startup.
  b) Create a new smolagents Tool subclass SearchSimilarQuotes
     that calls a vector store's similarity_search() method.
  c) Replace LookupPastQuotesTool in the quote agent's tool
     list with this new semantic search tool.
 
IMPROVEMENT 2 — Deterministic Discount Validation Tool
--------------------------------------------------------------
The test run revealed that the Quote Agent occasionally applies
discounts inconsistently — citing a discount on orders below
the $500 threshold or applying the wrong tier. This happens
because the discount logic is expressed only in the system
prompt, leaving the LLM to reason about it rather than
enforcing it mechanically.
 
The fix is to create a dedicated CalculateDiscountTool that
computes the correct discounted total deterministically:
 
  def forward(self, raw_total: float) -> str:
      if raw_total > 5000:   rate = 0.15
      elif raw_total > 1000: rate = 0.10
      elif raw_total > 500:  rate = 0.05
      else:                  rate = 0.0
      return json.dumps({"raw_total": raw_total,
                         "discount_rate": rate,
                         "discounted_total": raw_total*(1-rate)})
 
The Quote Agent would be required to call this tool for every
quote rather than computing discounts itself. This removes the
LLM reasoning step from a purely computational task, eliminates
the inconsistency observed in the test results, and makes
discount calculations fully auditable and reproducible.
