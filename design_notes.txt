1. SYSTEM OVERVIEW
----------------------------------------------------------------
 
The Munder Difflin Multi-Agent System automates three core
business workflows for a fictional paper supply company:
 
  1. Inventory management and restocking
  2. Quote generation with bulk discounts
  3. Order fulfillment and transaction recording
 
The system processes incoming customer requests end-to-end,
from checking stock through to recording revenue, using a
pipeline of four coordinated AI agents.
 
----------------------------------------------------------------
2. AGENT ARCHITECTURE (4 Agents)
----------------------------------------------------------------
 
AGENT 1 — Inventory Agent
  Role    : Check stock levels; restock items that are low
  Tools   : check_inventory, get_full_inventory, restock_item
  Inputs  : Customer request text, date
  Outputs : JSON summary of items checked and any restock orders
  Logic   : If any requested item has < 100 units OR less than
            the requested quantity, automatically places a
            restock order (min 500 units). Cash-aware: never
            orders more than the available cash balance allows.
 
AGENT 2 — Quote Agent
  Role    : Generate an itemised, discount-aware price quote
  Tools   : lookup_past_quotes, get_item_price, get_catalogue
  Inputs  : Customer request, date, inventory summary
  Outputs : JSON quote with line items, discount rate, total
  Logic   : Searches historical quote data for pricing context,
            maps customer descriptions to exact catalogue items
            (fuzzy matching), and applies tiered bulk discounts:
              - Orders > $500  → 5% discount
              - Orders > $1000 → 10% discount
              - Orders > $5000 → 15% discount
 
AGENT 3 — Order Agent
  Role    : Execute confirmed sale transactions
  Tools   : process_sale, get_cash_balance_tool,
            get_supplier_lead_time
  Inputs  : Customer request, date, approved quote
  Outputs : JSON transaction summary with statuses and revenue
  Logic   : Verifies cash availability, processes each line item
            as a separate sale transaction (one DB call per
            item), skips items with insufficient stock and
            reports them clearly.
 
AGENT 4 — Orchestrator
  Role    : Coordinate agents 1-3; synthesise customer response
  Tools   : None (reasoning and synthesis only)
  Inputs  : Customer request + outputs from all three agents
  Outputs : Plain-English customer-facing response
  Logic   : Runs Agents 1→2→3 in sequence, passing context
            forward at each step. Synthesises all results into
            a single professional response including quote
            details, fulfillment status, and delivery dates.
 
----------------------------------------------------------------
3. AGENT COMMUNICATION PATTERN
----------------------------------------------------------------
 
All agents communicate via structured text (JSON passed as
prompt context). There is no shared memory or message bus.
 
Pipeline flow per request:
 
  Customer Request
       │
       ▼
  [Inventory Agent] ──→ Inventory Summary (JSON)
       │
       ▼
  [Quote Agent] ──→ Quote Details (JSON)
       │
       ▼
  [Order Agent] ──→ Transaction Summary (JSON)
       │
       ▼
  [Orchestrator] ──→ Customer Response (plain English)
 
Each agent receives the outputs of all previous agents as
context in its system/user prompt. This creates a chain where
each agent has full awareness of prior decisions.
 
----------------------------------------------------------------
4. TOOL DESIGN DECISIONS
----------------------------------------------------------------
 
Tool implementations (_check_inventory, _restock_item, etc.)
are thin Python wrappers around the database utility functions
provided in the starter code. Each tool:
 
  - Returns a JSON string (not a Python object) so it can be
    passed back to the LLM as a tool result
  - Validates constraints before committing (e.g. stock check
    before processing a sale, cash check before restocking)
  - Handles errors gracefully and returns informative error
    messages in the JSON response
 
The _get_item_price tool includes fuzzy matching: if the LLM
passes a description like "heavy cardstock" instead of the
exact catalogue name "250 gsm Cardstock", the tool finds the
closest match so quotes still succeed.
 
----------------------------------------------------------------
5. THE run_agent() FUNCTION
----------------------------------------------------------------
 
All four agents share a single reusable run_agent() function
that implements the standard OpenAI tool-use loop:
 
  1. Send system prompt + user message to the LLM
  2. If the response contains tool_calls, execute each one
  3. Append tool results to the message history
  4. Send updated messages back to the LLM
  5. Repeat until LLM produces a final text response (no tools)
  6. Return the final text
 
Max turns = 6 per agent (prevents infinite loops).
 
----------------------------------------------------------------
6. FRAMEWORK CHOICE RATIONALE
----------------------------------------------------------------
 
The requirements.txt provided with the project lists only:
  pandas, openai, SQLAlchemy, python-dotenv, typing
 
Since no agent orchestration framework (smolagents, pydantic-ai,
npcsh) was included, the system was built using the OpenAI SDK
directly with function calling (tool use). This is the most
portable approach and requires zero additional dependencies
beyond what was already specified.

10. FUTURE IMPROVEMENT SUGGESTIONS
----------------------------------------------------------------
 
IMPROVEMENT 1 — Persistent Agent Memory with Quote Embeddings
--------------------------------------------------------------
Currently the quote_agent searches historical quotes using simple
keyword matching via search_quote_history(). A more powerful
approach would be to embed each historical quote as a vector
(using OpenAI's text-embedding-3-small model) and store them in
a vector database (e.g. ChromaDB or pgvector). When a new
customer request arrives, the quote_agent would embed the request
and retrieve the top-K most semantically similar past quotes —
not just keyword matches. This would improve pricing accuracy for
complex or unusual requests where keyword overlap is low (e.g.
a request for "biodegradable event supplies" would find quotes
for "eco paper plates" even without exact keyword overlap).
 
Implementation steps:
  a) Add an embedding step to init_database() that vectorises
     all rows in the quotes table on startup.
  b) Create a new smolagents tool SearchSimilarQuotesTool that
     calls a vector store's similarity_search() method.
  c) Replace the current lookup_past_quotes tool in the quote
     agent's tool list with this new semantic search tool.
 
IMPROVEMENT 2 — Retry Logic and Circuit Breaker for Supplier Orders
--------------------------------------------------------------------
The current restock_item tool makes a single attempt to record a
stock order. In production, two failure modes are common: (a) the
LLM API call itself times out or returns a rate-limit error, and
(b) the database transaction fails due to a connection issue.
Neither case is currently handled — the agent simply fails
silently or raises an unhandled exception.
 
A robust implementation would add:
 
  a) Retry logic in the RestockItemTool.forward() method using
     exponential backoff (e.g. tenacity library) — retry up to 3
     times on transient DB errors before surfacing a failure.
 
  b) A circuit breaker pattern in the ToolCallingAgent wrapper:
     if the same tool raises errors on 3 consecutive calls within
     one agent run, mark it as "unavailable" for the remainder of
     that run and have the agent fall back gracefully (e.g. log
     the restock intent without committing a transaction).
 
  c) Idempotency keys on create_transaction() calls so that if a
     network retry causes a duplicate call, the DB layer detects
     and deduplicates the transaction rather than recording it
     twice (which would double-debit cash).
 
These two improvements together would move the system from
prototype-grade to production-grade reliability.
